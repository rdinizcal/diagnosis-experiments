# RUNBOOK — diagnosis-experiments

Procedure for the repository owner to publish this repository and drive the
evaluation batch on GitHub Actions. Nothing here has been pushed yet.

## 0. One-time authorship check (run before pushing)

```bash
cd diagnosis-experiments
git log --format='%an <%ae> | %cn <%ce>' | sort -u        # exactly one identity
git log | grep -inE 'claude|codex|copilot|anthropic|co-authored|generated'   # empty
git grep -inE 'claude|codex|anthropic|as an ai|llm|generated by' -- ':!*.lock' # empty
```

## 1. Create the public repo and push

```bash
# Create the empty public repo (no README/license so history stays clean):
gh repo create <owner>/diagnosis-experiments --public --disable-wiki

# From this directory (already `git init`ed with one commit):
git remote add origin https://github.com/<owner>/diagnosis-experiments.git
git branch -M main
git push -u origin main
```

## 2. Enable Actions

```bash
gh api -X PUT repos/<owner>/diagnosis-experiments/actions/permissions \
  -F enabled=true -f allowed_actions=all   # -F sends enabled as a boolean
```
(or Settings → Actions → General → Allow all actions, in the web UI.)

## 3. Canary run (do this first)

Confirms the toolchain — Python deps, Weka download + checksum, one real cell —
before spending the full matrix.

```bash
gh workflow run batch.yml -f seeds=0 -f subjects=AT1,AT2 -f wall_minutes=330
gh run watch $(gh run list --workflow=batch.yml -L1 --json databaseId -q '.[0].databaseId')
```

If the **Fetch Weka** step fails on the checksum, the Maven artifact differs
from the pinned hash: download it, recompute, and update `WEKA_SHA256` in both
`.github/workflows/batch.yml` and `smoke.yml` (see "Weka checksum" below).

## 4. Full batch

```bash
gh workflow run batch.yml -f seeds=0 -f subjects=all -f wall_minutes=330
# multiple seeds:
gh workflow run batch.yml -f seeds=0,1,2,3,4 -f subjects=all -f wall_minutes=330
```

Matrix = subjects x seeds, `max-parallel: 20`, `timeout-minutes: 355` (under the
6-hour hosted-runner cap). Each cell runs the subject's exp1 + exp3 under an
internal wall guard (`wall_minutes`) that leaves a resumable state before the
runner cap; the SQLite verdict cache (restored via `actions/cache`) makes a
re-run of the same cell resume rather than restart.

## 5. Aggregate

```bash
BATCH_ID=$(gh run list --workflow=batch.yml -L1 --json databaseId -q '.[0].databaseId')
gh workflow run aggregate.yml -f run_id=$BATCH_ID
```
`aggregate.yml` also triggers automatically when a batch run completes. It writes
the summary to the job summary and uploads the `diagnosis-statistics` artifact
(CSV per table + `SUMMARY.md`).

## 6. Pull results locally

```bash
gh run download <batch_run_id>       # all diagnosis-run-* artifacts
gh run download <aggregate_run_id>   # diagnosis-statistics
```

## Notes

- **Determinism.** The seed comes only from the config. `batch.yml` runs with
  `--workers 1` (serial), so nothing varies with the runner; worker count does
  not affect results in any case (batch results are applied in population-index
  order). Runner hardware is recorded in each `report.json` under `runner`.
- **Artifacts.** Each `diagnosis-run-*` contains `report.json`, `summary.json`,
  `run_meta.json`, final ARFFs, J48 `.out`, stopping/`hypot` logs, and
  `inference_state.json`. Bulky per-generation population dumps are excluded.
  Retention: 30 days.
- **Re-dispatch is a fresh run.** There is no cross-dispatch resume: the verdict
  cache lives inside each timestamped run dir, so re-running a cell recomputes it
  from scratch. Runs are bounded by `max_samples` / `cv_pr` stopping, so they
  terminate; a wall-stopped cell is simply re-run in full. (Aggregation keeps
  only the newest run per subject/seed, so a re-run supersedes an earlier one.)
- **Weka needs bounce.jar.** The Maven `weka-stable-3.8.6.jar` does not bundle
  `org.bounce`, and Weka's package manager references it, so J48 dies with
  `NoClassDefFoundError` and produces no tree unless `third_party/bounce.jar`
  (vendored) is on `WEKA_JAR`. The workflows add it automatically; keep it when
  editing the Weka step. See `third_party/README.md`.
- **Weka checksum.** To refresh:
  ```bash
  curl -fsSL -o weka.jar \
    https://repo1.maven.org/maven2/nz/ac/waikato/cms/weka/weka-stable/3.8.6/weka-stable-3.8.6.jar
  sha256sum weka.jar
  ```
- **Solve-heavy subjects (CC).** CC requirements can need the 600 s high tier
  even locally; on slower runners those SAT solves time out to UNDECIDED, so a
  CC cell may show 0 SAT. Raise `two_tier_timeout.high_sec` (<= trace timeout)
  in `scripts/gen_batch_configs.py` for CI, or run CC locally with no wall guard.
- **Timing.** No timing claim in the paper is derived from CI.
